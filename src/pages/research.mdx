## 1. Executive Summary

Stagent is the **coordination layer for the agentic enterprise**: a local-first operations workspace where AI agents run inside a governed system of projects, workflows, documents, inbox approvals, profiles, schedules, live monitoring, and cross-runtime cost governance.

The thesis is simple. Every enterprise will deploy AI agents. But production adoption will be constrained less by model quality than by whether operators can see what the agent is doing, understand which rules it is following, reuse proven behaviors, and intervene before an unsafe action lands. Stagent is built for that operating problem.

Four value propositions define Stagent's current position:

1. **Operational familiarity** — Stagent presents agent work through surfaces teams already understand: a home workspace, execution dashboard, project views, document library, approval inbox, and live monitor.
2. **Governed execution** — The human-in-the-loop model is not an afterthought. Tool requests, policy decisions, and intervention points are visible, auditable, and close to the work.
3. **Reusable automation primitives** — Profiles, workflow blueprints, and schedules convert ad hoc prompting into repeatable operational systems.
4. **Cost accountability** — A shared usage ledger, spend guardrails, and a dedicated cost dashboard turn provider choice into something teams can supervise rather than guess at.

As of March 2026, the current browser product spans **eleven coordinated operator surfaces**: Home, Dashboard, Projects, Workflows, Documents, Profiles, Schedules, Inbox, Monitor, Cost & Usage, and Settings. It is a full AI operations workspace rather than a narrow single-purpose execution screen.

### The Governance Imperative

The numbers reveal a paradox. Enterprises increasingly believe agents can produce output, but many still cannot convert those gains into durable operating leverage. The bottleneck is no longer access to a model. It is the ability to govern model behavior in real workflows.

This problem intensifies as regulatory pressure increases. The EU AI Act takes full effect in August 2026, and organizations deploying agents into real operational processes will need visible oversight, clear intervention paths, and better auditability than a chat log can provide.

## 4. Product Architecture

### Design Philosophy

Stagent's architecture follows three principles:

1. **Simplicity** — Prefer a local-first, legible stack over unnecessary infrastructure sprawl.
2. **Transparency** — Every execution should be reviewable through product surfaces built for operators, not just developers.
3. **Governed autonomy** — Let agents work at speed, but keep trust boundaries explicit and interruptible.

### The canUseTool Governance Pattern

The centerpiece of Stagent's architecture remains the `canUseTool` pattern: the mechanism that converts autonomous execution into governed execution without hiding what the agent is doing.

<Figure number={4} caption="canUseTool Governance Flow — 5-Step Human-in-the-Loop Cycle">
  <GovernanceFlow />
</Figure>

<Figure number={5} caption="Permissions Inbox — Explicit Approval Controls for Agent Tool Requests">
  <BrowserFrame src="/screenshots/inbox-permissions.png" alt="Permissions inbox showing allow once, always allow, and deny actions" />
</Figure>

1. **Agent requests a tool** — The runtime reaches a trust boundary.
2. **Inbox item is created** — Stagent records the request with enough detail for review.
3. **Human reviews the action** — The operator decides whether the request is safe and appropriate.
4. **Decision is written back** — Allow, always allow, or deny becomes part of the system state.
5. **Execution resumes or stops** — The agent proceeds according to the human decision.

### Reusable Automation Layer

What changed materially in the current product is the addition of reusable automation primitives:

- **Profiles** make instructions, tool policy, and runtime tuning inspectable and reusable.
- **Workflow blueprints** reduce orchestration setup time by turning common operational patterns into templates.
- **Schedules** let teams move from one-off prompting to recurring, guardrailed automation.

These primitives matter because they turn a successful experiment into an operational practice. Without them, every agent run is effectively a bespoke interaction. With them, the organization begins building an operating system for repeated AI work.

### Cost & Usage Governance

What changed next is that Stagent now measures and governs the economics of that automation layer instead of treating spend as invisible exhaust.

- **Usage ledger** normalizes Claude and Codex activity into one cost and token record system.
- **Budget guardrails** warn near caps and block new paid activity explicitly when a configured window is exhausted.
- **Cost & Usage dashboard** shows spend velocity, token velocity, runtime share, and model concentration in one operator-facing surface.

## Appendix A: Technical Glossary

| Term | Definition |
|------|-----------|
| **Agent Runtime** | The governed execution backend used to run Claude Agent SDK or Codex App Server sessions inside Stagent. |
| **Blueprint** | Preconfigured workflow template that bootstraps a new multi-step automation. |
| **canUseTool** | Callback invoked when the runtime reaches a tool-use trust boundary. |
| **Drizzle ORM** | TypeScript-first ORM used for local database access. |
| **HITL** | Human-in-the-loop decision point requiring explicit operator approval or input. |
| **Profile** | Reusable agent definition containing instructions, tool policy, and tuning settings. |
| **Schedule** | Recurring or one-shot automation record that generates work on a cadence. |
| **SSE** | Server-Sent Events used for live monitor streaming. |
| **SQLite WAL** | Write-Ahead Logging mode supporting concurrent reads in a local database. |
| **Usage Ledger** | Normalized cross-runtime record of tokens, cost, provider, model, status, and execution linkage. |
